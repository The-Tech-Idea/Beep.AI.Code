# 1. Harden the agent loop first

Do this before vector DB, test generation, or fancy subagents.

Your loop should look like this:

```text
User task
→ classify task
→ inspect repo
→ select context
→ plan
→ execute tool
→ inject structured result/error
→ repair
→ validate
→ summarize diff
```

You likely already have something like this, but you need to make the loop more explicit.

Suggested files:

```text
beep/
  agent/
    loop.py
    state.py
    planner.py
    executor.py
    validator.py
    retry_policy.py
```

Core state model:

```python
from dataclasses import dataclass, field
from enum import Enum
from typing import Any


class AgentPhase(str, Enum):
    UNDERSTAND = "understand"
    CONTEXT = "context"
    PLAN = "plan"
    EXECUTE = "execute"
    VALIDATE = "validate"
    REPAIR = "repair"
    FINAL = "final"


@dataclass
class AgentState:
    task: str
    phase: AgentPhase = AgentPhase.UNDERSTAND
    steps: list[dict[str, Any]] = field(default_factory=list)
    failures: list[dict[str, Any]] = field(default_factory=list)
    changed_files: set[str] = field(default_factory=set)
    test_results: list[dict[str, Any]] = field(default_factory=list)
    build_results: list[dict[str, Any]] = field(default_factory=list)
    max_iterations: int = 25
    iteration: int = 0

    def add_step(self, kind: str, data: dict[str, Any]) -> None:
        self.steps.append({"kind": kind, "data": data})

    def add_failure(self, failure: dict[str, Any]) -> None:
        self.failures.append(failure)
```

Your current anti-loop guards are good, but you need **phase-aware control**. For example:

```text
If editing happened → validation is mandatory.
If tests fail → repair phase is mandatory.
If same command fails twice → inject diagnosis request.
If no file changed after N edit attempts → force re-plan.
```

This is one of the most important missing pieces.

Right now you have tool results, shell exit codes, and feedback. Good. But the model needs errors in a normalized format.

Create this:

```text
beep/errors/

```

Example:

```python
from dataclasses import dataclass
from typing import Optional


@dataclass
class StructuredToolError:
    tool_name: str
    error_type: str
    message: str
    command: Optional[str] = None
    file_path: Optional[str] = None
    line: Optional[int] = None
    column: Optional[int] = None
    exit_code: Optional[int] = None
    stdout: Optional[str] = None
    stderr: Optional[str] = None
    retryable: bool = True
    suggested_action: Optional[str] = None
```

Use categories like:

```text
BuildError
TestFailure
CommandTimeout
PermissionDenied
FileNotFound
InvalidPatch
McpToolError
MissingDependency
SyntaxError
RuntimeError
UnknownTool
RepeatedFailure
```

Then inject it into the next model message like this:

```xml
<tool_error>

  <error_type>BuildError</error_type>

  <exit_code>1</exit_code>
  <stderr>...</stderr>
  <suggested_action>Fix the compile error before continuing.</suggested_action>
</tool_error>
```

This is much better than only giving raw stderr.

# 3. Add an edit transaction system

Your agent should not just write files. It should edit through a transaction layer.

Suggested files:

```text
beep/editing/
  patcher.py
  transaction.py
  diff.py
  rollback.py
```

You need this behavior:

```text
Before edit:
  snapshot file

Apply edit:
  patch or full rewrite

After edit:
  generate diff
  record changed file

If validation fails badly:
  allow rollback

Final:
  show changed files and summary
```

Example:

```python
from dataclasses import dataclass
from pathlib import Path


@dataclass
class FileSnapshot:
    path: Path
    original: str


class EditTransaction:
    def __init__(self) -> None:
        self.snapshots: dict[Path, FileSnapshot] = {}
        self.changed_files: set[Path] = set()

    def snapshot(self, path: Path) -> None:
        if path not in self.snapshots and path.exists():
            self.snapshots[path] = FileSnapshot(path=path, original=path.read_text(encoding="utf-8"))

    def write_text(self, path: Path, content: str) -> None:
        self.snapshot(path)
        path.write_text(content, encoding="utf-8")
        self.changed_files.add(path)

    def rollback(self) -> None:
        for snapshot in self.snapshots.values():
            snapshot.path.write_text(snapshot.original, encoding="utf-8")
```

This is a big quality upgrade.

# 4. Add a real context manager

This is probably your biggest future differentiator.

Right now you have semantic search via Semble. That is useful, but a coding agent needs **hybrid context**, not only vector search.

Implement:

```text
beep/context/
  manager.py
  repo_map.py
  symbol_index.py
  file_ranker.py
  context_budget.py
  dependency_graph.py
```

Your context manager should combine:

```text
1. User-mentioned files
2. Recently edited files
3. Semantic search results
4. Text search results
5. Symbol references
6. Import/dependency graph
7. Test files related to source files
8. Project rules from .beep.md
```

For Python codebases, use:

```text
ast
tree-sitter
ripgrep
import graph
pytest discovery
```

For C#/.NET projects, you can later add Roslyn through external tooling, but for your Python agent core, keep the context manager language-agnostic.

A useful ranking model:

```python
@dataclass
class ContextCandidate:
    path: str
    score: float
    reason: str
    source: str
```

Example scoring:

```text
+100 explicitly mentioned by user
+80 recently changed by agent
+70 contains symbol name
+60 semantic match
+50 imported by target file
+40 test file for target file
-100 ignored/generated/vendor file
```

This is more important than adding Chroma/Qdrant immediately.

# 5. Add automatic test generation as a dedicated capability

You are correct: the agent can manually write tests today, but a good coding agent should have a **test-generation workflow**.

Do not implement it as just another prompt. Implement it as a workflow/tool.

Suggested files:

```text
beep/testing/
  generator.py
  planner.py
  coverage.py
  test_file_locator.py
```

Tool idea:

```python
@dataclass
class GenerateTestsRequest:
    target_files: list[str]
    framework: str | None = None
    behavior_description: str | None = None
    max_tests: int = 5


@dataclass
class GenerateTestsResult:
    test_files: list[str]
    created_files: list[str]
    updated_files: list[str]
    command_to_run: str
```

Workflow:

```text
Find target source file
Find existing test style
Detect framework
Generate tests
Run tests
If tests fail because test is wrong → repair test
If tests reveal bug → ask/continue based on task
```

Important: generated tests should follow existing project style. The agent should first inspect:

# 7. Add a sandbox and permission policy

If your agent can run shell commands and edit files, you need a real policy layer.

Suggested files:

```text
beep/security/
  policy.py
  sandbox.py
  approvals.py
  command_guard.py
  path_guard.py
```

Minimum policy:

```python
from dataclasses import dataclass, field


@dataclass
class PermissionPolicy:
    allow_file_write: bool = True
    allow_shell: bool = True
    allow_network: bool = False
    require_approval_for_shell: bool = False
    require_approval_for_destructive: bool = True
    protected_paths: list[str] = field(default_factory=lambda: [
        ".git",
        ".env",
        "secrets.json",
        "id_rsa",
        "id_ed25519",
        "node_modules",
        ".venv",
        "venv",
        "__pycache__",
    ])
    blocked_commands: list[str] = field(default_factory=lambda: [
        "rm -rf /",
        "sudo",
        "mkfs",
        "shutdown",
        "reboot",
    ])
```

Before every command:

```text
Classify command risk
Check working directory
Check protected paths
Check network usage
Ask approval if needed
Run with timeout
Capture stdout/stderr
Return structured result
```

Add:

```text
beep/validation/
  policy.py
  build.py
  tests.py
  lint.py
  typecheck.py
```

Validation policy should be project-specific.

Example:

```python
@dataclass
class ValidationPolicy:
    run_tests_after_edit: bool = True
    run_lint_after_edit: bool = False
    run_typecheck_after_edit: bool = False
    max_repair_attempts: int = 3
    required_commands: list[str] = None
```

The agent should infer common commands:

```text
Python:
  pytest
  ruff check .

Node:
  npm test
  npm run build
  npm run lint

.NET:
  dotnet build
  dotnet test

Rust:
  cargo test

  go test ./...
```

Final answer should say:

```text

- x
- y

- pytest passed
```

This makes the agent feel trustworthy.

If you want the “best” coding agent, you need to measure it.

Suggested files:


```yaml
name: fix_python_import_error
repo: samples/import_error_project
task: "Fix the failing tests."
success:
  commands:
    - "pytest"
  changed_files_max: 3
```

Metrics:

```text
Task success
Build success
Test success
Number of tool calls
Number of failed tool calls
Number of changed files
Did it edit protected files?
Did it create unnecessary files?
Did it explain final result correctly?
```

This is how you improve the agent scientifically instead of guessing.
