# 4. Python code editing/refactoring

Use **LibCST** for Python-specific edits.

LibCST parses Python into a concrete syntax tree that preserves formatting details like comments, whitespace, and parentheses. ([libcst.readthedocs.io][8]) Its codemod framework is specifically designed for automated refactors across large codebases. ([libcst.readthedocs.io][9])

Install:

```bash id="fb4sw5"
pip install libcst
```

Use it for:

```text id="ruaxum"
safe imports editing
renaming functions/classes
adding methods
changing function signatures
moving code
automated refactors
```

For example, if the agent wants to add an import, avoid string hacks. Use LibCST.

```python id="03374k"
import libcst as cst
from libcst import codemod


class AddImportTransformer(cst.CSTTransformer):
    def leave_Module(self, original_node: cst.Module, updated_node: cst.Module) -> cst.Module:
        new_import = cst.SimpleStatementLine(
            body=[
                cst.ImportFrom(
                    module=cst.Name("typing"),
                    names=[cst.ImportAlias(cst.Name("Protocol"))],
                )
            ]
        )

        return updated_node.with_changes(
            body=[new_import, *updated_node.body]
        )
```

For Python code, **LibCST is better than raw text edits**.

# 5. Clean Code and architecture enforcement

Use **Ruff** and **Import Linter**.

Ruff is a Python linter and formatter with `pyproject.toml` support, automatic fixes, built-in caching, and many built-in rules. ([Astral Docs][10])

Install:

```bash id="z86hym"
pip install ruff
```

Use:

```bash id="alwr52"
ruff check .
ruff format .
```

For DDD/Clean Architecture, use **import-linter**.

Import Linter lets you impose constraints on imports between Python modules, which is exactly what you need for enforcing layers. ([import-linter.readthedocs.io][11]) It supports forbidden contracts, such as preventing one module group from importing another. ([import-linter.readthedocs.io][12])

Install:

```bash id="bct7pz"
pip install import-linter
```

Example `.importlinter`:

```ini id="8q654z"
[importlinter]
root_package = my_app

[importlinter:contract:domain_must_not_import_infrastructure]
name = Domain must not import infrastructure
type = forbidden
source_modules =
    my_app.domain
forbidden_modules =
    my_app.infrastructure
    my_app.presentation

[importlinter:contract:application_must_not_import_presentation]
name = Application must not import presentation
type = forbidden
source_modules =
    my_app.application
forbidden_modules =
    my_app.presentation
```

Then your agent can run:

```bash id="wwa6ef"
lint-imports
```

This is very useful for your goal: **clean code + domain services + layered architecture**.

# 7. Persistent memory / vector DB

Since you already use Semble, you do not have to rush this. But if you want persistent vector memory:

## Best local-first choices

| Library     | Best for                                                          |
| ----------- | ----------------------------------------------------------------- |
| **Chroma**  | Simple local vector DB for RAG/prototypes                         |
| **Qdrant**  | Strong production-ready vector search, local or server            |
| **LanceDB** | Local-first embedded vector store, good for code/document indexes |

Chroma’s docs describe it as retrieval infrastructure with embeddings, metadata filtering, dense/sparse search, and retrieval. ([Chroma Docs][13]) Qdrant’s Python client supports sync and async requests to Qdrant. ([Qdrant Python Client][14]) LanceDB’s Python API supports sync and async connections for creating/opening tables and querying/modifying data. ([LanceDB][15])

My recommendation:

```text id="sjjnqd"
Use Chroma for fastest prototype.
Use Qdrant if you want production/server mode.
Use LanceDB if you want embedded local project indexes.
```

For your coding agent, I would pick **LanceDB** or **Qdrant local mode**.
