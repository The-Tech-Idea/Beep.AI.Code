# 1. For multi-language parsing: `tree-sitter`

Use this when you want your agent to understand many languages.

`tree-sitter` can build concrete syntax trees and update them efficiently as source code changes. The Python binding is available as `tree-sitter`. ([GitHub][1])

Good for:

```text
class/function extraction
imports/includes/usings
method boundaries
symbol outlines
finding changed functions
multi-language context selection
```

Recommended module:

```text
beep/codeanalysis/
  tree_sitter_analyzer.py
```

Example model:

```python
from dataclasses import dataclass


@dataclass
class CodeSymbol:
    name: str
    kind: str
    file_path: str
    start_line: int
    end_line: int
```

Use `tree-sitter` for the **general code map** of the project.

---

# 2. For Python code analysis/editing: `LibCST`

Use `libcst` when your agent edits Python code.

LibCST parses Python source into a concrete syntax tree that keeps formatting details such as comments, whitespace, and parentheses, which makes it useful for codemods and linters. ([libcst.readthedocs.io][2])

Good for:

```text
adding imports safely
renaming functions/classes
moving code
adding decorators
refactoring function signatures
building Python-specific lint rules
```

This is better than string editing for Python.

Example use:

```python
import libcst as cst


class FunctionCollector(cst.CSTVisitor):
    def __init__(self):
        self.functions: list[str] = []

    def visit_FunctionDef(self, node: cst.FunctionDef) -> None:
        self.functions.append(node.name.value)
```

Use `LibCST` for **Python-aware analysis and safe Python edits**.

---

# 3. For Python semantic inference: `astroid`

Use `astroid` when you need deeper Python static analysis than the built-in `ast` module.

Astroid is a library for AST parsing, static analysis, and inference, and it powers much of Pylint’s analysis. ([pylint.pycqa.org][3])

Good for:

```text
inferring object types
finding class attributes
understanding imports
analyzing function calls
detecting possible member access issues
```

Use `astroid` for **Python semantic analysis**.

---

# 4. For Python references and navigation: `jedi`

Use `jedi` if your agent needs IDE-like behavior.

Jedi is a Python static analysis library focused on autocompletion, goto functionality, refactoring, code search, and finding references. ([jedi.readthedocs.io][4])

Good for:

```text
go to definition
find references
list names in a file
autocomplete-like suggestions
understand imported symbols
```

Use `jedi` for **repo navigation and symbol references**.

---

# 5. For lint/style analysis: `ruff`

Use `ruff` as the default Python linter/formatter.

Ruff is an extremely fast Python linter and code formatter written in Rust, and its formatter is designed as a drop-in replacement for Black. ([Astral Docs][5])

Good for:

```text
style issues
unused imports
bad patterns
formatting
import sorting
automatic fixes
```

Your agent should run:

```bash
ruff check .
ruff format .
```

or safer:

```bash
ruff check . --fix
ruff format .
```

---

# 6. For type analysis: `mypy` or `pyright`

Use one of these for Python type checking.

`mypy` is a static type checker that finds type-related bugs without running the program. ([mypy Documentation][6])

`pyright` is a full-featured, standards-based static type checker designed for high performance and large Python codebases. ([GitHub][7])

Use:

```bash
mypy .
```

or:

```bash
pyright
```

For your agent, I would support both and detect which one the project already uses.

---

# 7. For complexity and maintainability: `radon`

Use `radon` to detect complex code.

Radon can compute raw metrics, cyclomatic complexity, Halstead metrics, and maintainability index, and it also exposes an API. ([Radon][8])

Good for:

```text
large/complex functions
classes that need refactoring
maintainability scores
before/after refactor comparison
```

Commands:

```bash
radon cc .
radon mi .
```

Use this for your **Clean Code reviewer**.

---

# 8. For architecture and dependency graphs: `grimp` + `import-linter`

Use `grimp` to build import graphs. Grimp provides an `ImportGraph` API representing imports inside Python packages. ([Grimp][9])

Use `import-linter` to enforce layer rules. Import Linter lets you impose constraints on imports between Python modules, including forbidden dependencies between module groups. ([Import Linter][10])

This is perfect for your goal:

```text
domain must not import infrastructure
domain must not import presentation
application must not import presentation
infrastructure may depend on application/domain
presentation may depend on application
```

Example `.importlinter`:

```ini
[importlinter]
root_package = my_app

[importlinter:contract:domain_no_infra]
name = Domain must not import infrastructure
type = forbidden
source_modules =
    my_app.domain
forbidden_modules =
    my_app.infrastructure
    my_app.presentation

[importlinter:contract:application_no_presentation]
name = Application must not import presentation
type = forbidden
source_modules =
    my_app.application
forbidden_modules =
    my_app.presentation
```

Then run:

```bash
lint-imports
```

This is one of the best tools for **DDD / Clean Architecture enforcement** in Python.

---

# 9. For security/custom static rules: `semgrep`

Use `semgrep` for custom rules and security scanning.

Semgrep is a static analysis tool for finding bugs, vulnerabilities, and enforcing coding standards; it supports many languages and custom rules. ([Semgrep][11])

Good for:

```text
security issues
forbidden patterns
custom clean-code rules
custom architecture rules
multi-language static analysis
```

Example use:

```bash
semgrep --config auto .
```

You can also write rules like:

```yaml
rules:
  - id: no-business-logic-in-controller
    pattern: |
      class $XController:
          ...
    message: "Controller should not contain business logic."
    severity: WARNING
    languages: [python]
```

# 10. For C#/.NET analysis from your Python agent

For C#, do **not** rely only on Python libraries.

Use your Python agent to call a small .NET/Roslyn analyzer tool.

Roslyn APIs let tools examine and understand C# and Visual Basic codebases using syntax and semantic analysis APIs, and they can build analyzers, code fixes, and refactorings. ([Microsoft Learn][12])

Recommended approach:

```text
Python agent
  → calls dotnet analyzer CLI
      → Roslyn analyzes .sln/.csproj
      → returns JSON symbols/errors/dependencies
  → Python agent uses JSON as context
```

Example output from your future Roslyn helper:

```json
{
  "symbols": [
    {
      "name": "OrderService",
      "kind": "class",
      "file": "Application/Orders/OrderService.cs",
      "methods": ["CreateOrder", "CancelOrder"]
    }
  ],
  "diagnostics": [
    {
      "file": "Domain/Order.cs",
      "line": 42,
      "code": "CS8602",
      "message": "Dereference of a possibly null reference."
    }
  ]
}
```

This is the right design for your .NET background.
